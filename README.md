# WebAPI для приема, парсинга и обработки CSV-файлов.  

## Описание реализации
Архитектура проекта построена на принципах **Clean Architecture** и **CQRS**. Все входящие запросы проходят через **Middleware**, который обеспечивает глобальную обработку исключений.

Все операции обработки файлов инкапсулированы в обработчиках команд (`Handlers`). Входящие данные проходят строгую валидацию на этапе парсинга.

Ключевые особенности реализации:
*   **Асинхронность:** Все операции ввода-вывода (чтение файла, работа с БД) реализованы через `async/await` для высокой производительности.
*   **Unit of Work:** Использование паттерна `IUnitOfWork` позваляет выполнять моножество операций связанных с изменением данных одной транзакцией
*   **Идемпотентность:** При повторной загрузке файла с тем же именем система автоматически удаляет старые записи перед сохранением новых (**Это происходит в рамках одной транзакции!**).
*   **Типизация:** Использование `Guid` для первичных ключей и DTO для передачи данных между слоями.

Взаимодействие между слоями построено на основе абстракций, что обеспечивает высокую тестируемость и возможность легкой замены реализаций.

## Функционал

- **Загрузка CSV:** Асинхронный парсинг файлов.
- **Валидация данных:** 
    - Дата не может быть позже текущей и раньше 01.01.2000
    - Время выполнения не может быть меньше 0
    - Значение показателя не может быть меньше 0
    - Количество строк не может быть меньше 1 и больше 10 000
    - Значения должны соответствовать своим типам, отсутствие одного из значений в записи недопустимо
- **Расчет статистики:**
    - Дельта времени Date в секундах
    - Минимальное дата и время, как момент запуска первой операции
    - Среднее время выполнения
    - Среднее значение по показателям
    - Медина по показателям
    - Максимальное значение показателя
    - Минимальное значение показателя
- **Хранение данных:** В качестве СУБД используется PostgreSQL, а взаимодействие происходит через EF Core.
- **Покрытие тестами:** Unit-тесты.

## Технологии

- **Язык**: C# (.NET 8)

## Запуск проекта

### 1. Подготовка базы данных
Если папка `Migrations` пуста, создайте начальную миграцию:

```bash
dotnet ef migrations add InitialCreate
```

### 2. Сборка и запуск через Docker Compose
Убедитесь, что Docker запущен, и выполните:

```bash
docker compose up --build
```

### 3. После запуска:
- **WebAPI** будет доступен на: `http://localhost:8080/swagger/index.html`

---

## Конфигурация

Строка подключения к БД задается через переменные окружения в `docker-compose.yml`:

```env
ConnectionStrings__DefaultConnection: "Host=db;Database=TimescaleDb;Username=TimescaleWebAPI;Password=Password"
```

## Тестирование

Проект разделен набор тестов:
- **WebAPI.UnitTests:** Юнит-тесты для проверки логики хендлеров, валидации и расчета статистики.

Для запуска всех тестов:
```bash
dotnet test
```

## Пример структуры данных

### Входной CSV файл
Файл должен содержать столбцы для даты, времени выполнения и значения.

```csv
Date;ExecutionTime;Value
2025-01-15T12:00:00.0000Z;1.5;123.45
2025-06-20T12:00:00.0000Z;2.3;456.78
2026-01-10T12:00:00.0000Z;0.5;789.01
2026-07-17T12:00:00.0000Z;3.2;234.56
2024-12-31T12:00:00.0000Z;1.0;567.89
```

## Обработка ошибок бизнес логики

Система использует паттерн `Result` для обработки ошибок без исключений в контроллерах.

Все ошибки имеют слои информации. Получение более подробной информации осуществляется через свойство `Details`.
